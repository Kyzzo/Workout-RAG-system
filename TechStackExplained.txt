Tech Stack Explained — Notes
=============================
What each piece of the stack is, what it does, and how it fits this project.


PACKAGE MANAGER — uv
---------------------
Installs and tracks the external libraries your code depends on (FastAPI,
SQLAlchemy, the OpenAI SDK, etc.) so you don't manually download and wire
them in yourself.

Two jobs:
1. Installs dependencies — you declare "I need FastAPI, SQLAlchemy, openai"
   in a project file, and it fetches them.
2. Pins exact versions (via a lockfile) — so your project runs identically
   on your machine, a teammate's machine, or a deployed server. Without
   this, "works on my machine" bugs happen constantly, because library
   version differences change behavior.

Also creates an isolated environment for the project, so this project's
library versions don't clash with some other Python project on your
machine that needs different versions of the same library.

uv is a newer, fast Python package manager (alternative to older tools
like pip/poetry) — already used in the RAG project (`uv sync` installs
everything listed in the project's config file).


ORM — SQLAlchemy
-----------------
Lets you interact with your database using Python classes/objects
(Program, program.goal) instead of writing raw SQL strings by hand.
SQLAlchemy translates your Python code into SQL behind the scenes.

Without an ORM:
    cursor.execute("SELECT * FROM program WHERE user_id = %s", (user_id,))
    rows = cursor.fetchall()   # raw tuples, manually mapped

With the ORM:
    programs = session.query(Program).filter(Program.user_id == user_id).all()
    # programs[0].goal works directly, as real Python objects

Short version: objects in, SQL out, automatically.


MIGRATIONS — Alembic
----------------------
Your Program/Mesocycle/etc. classes define what your database tables
SHOULD look like — but the database doesn't automatically stay in sync
with changes to those classes once real data exists in it.

A migration is a script describing one specific change to the database
structure ("add column X to table Y") that can be applied (and usually
reversed) — similar to how git tracks a history of code changes.

Alembic:
1. Compares your current SQLAlchemy models against the actual database's
   current state
2. Auto-generates a migration script describing the difference
3. Lets you apply it (alembic upgrade) to bring the database up to date

Without this, you'd hand-write ALTER TABLE SQL every time a model
changes — tedious and error-prone. Alembic automates and tracks it.


RELATIONAL DB — PostgreSQL
----------------------------
Stores structured data as tables with rows/columns and relationships
(foreign keys) between them — this is the actual schema (User, Program,
Mesocycle, etc.) as real, persisted data.

SQL = the query language. Postgres = the engine that runs it and stores
the data on disk.

Example raw query:
    SELECT * FROM program WHERE user_id = '123';


VECTOR DB — Qdrant
--------------------
A different kind of database, specialized for one job: finding
semantically similar chunks of text.

Instead of rows/columns, it stores embeddings — numeric vectors
representing meaning, generated by an embedding model (OpenAI's, here).
Asking "what does research say about training volume" finds study
chunks whose embeddings are mathematically closest to the question's
embedding — i.e. the most relevant text, even without shared keywords.

This is why there are TWO databases:
    Postgres -> structured facts (who owns what program)
    Qdrant   -> "find text similar in meaning to X" (RAG retrieval)
Different data shapes, different tools.


BACKGROUND JOBS — Inngest
----------------------------
Some tasks take too long to run inside a normal web request. Ingesting
a scientific PDF means: extract text -> split into chunks -> call OpenAI
to embed each chunk -> store each in Qdrant. For a long paper this could
take 30+ seconds — too slow for a request that should respond in under a
second, and if a step fails halfway through, the result is a half-broken
mess.

Background jobs let that work run separately from the request/response
cycle: the API responds immediately ("ingestion started"), and the real
multi-step work happens asynchronously.

Inngest is "durable" — it tracks each step of a multi-step job
individually. If the embedding step fails (e.g. an OpenAI API hiccup),
it retries just that step, not the whole pipeline. It also gives a
dashboard to see what ran, what failed, and why.

In this project: ingesting the scientific-study corpus is exactly this
scenario — long-running, multi-step, needs reliability.


AUTH — Clerk
-------------
Authentication = verifying who a user is (signup, login, sessions).
Building this correctly yourself is genuinely hard to get right securely
— password hashing, session tokens, password reset flows, email
verification, social login all have real security pitfalls if done wrong.

Clerk is a hosted service providing pre-built, secure login/signup UI
components dropped directly into the Next.js app, managing the
underlying complexity.

How it connects to the backend:
1. User logs in via Clerk's UI in the Next.js frontend
2. Clerk issues a JWT (JSON Web Token) — a signed, tamper-proof token
   saying "this is user X, verified, expires at time Y"
3. The frontend attaches that token to every API request to FastAPI
4. FastAPI verifies the token's signature (using Clerk's public key) to
   confirm the request is genuinely from that logged-in user

FastAPI never handles a password or manages sessions itself — Clerk does
the hard part, the backend just checks a signature.


BACKEND FRAMEWORK — FastAPI
------------------------------
Defines the actual API — the set of URLs (/programs, /mesocycles/{id})
the frontend calls, and the Python functions that run when they're hit.
Handles parsing incoming requests, validating data shapes, and
formatting responses. Routes "a request came in" to "here's the Python
code that handles it."


FRONTEND — Next.js + Tailwind CSS
------------------------------------
Next.js: the React-based framework rendering the actual UI — pages,
navigation, forms (intake form, program display). Frontend-only in this
architecture; it calls FastAPI for all real data/logic.

Tailwind CSS: styles that UI using utility classes
(className="p-4 bg-blue-500") instead of separate CSS files.


OPENAI
-------
Two distinct uses in this project:
1. Embeddings — turning text into vectors for Qdrant to search
2. Generation — the LLM call that synthesizes a structured program from
   retrieved study chunks

Same provider, two different API calls, two different jobs.


DEPLOYMENT — Vercel / Railway or Render / Qdrant Cloud
----------------------------------------------------------
Where each piece actually runs once development is done locally:
    Vercel          -> hosts the Next.js frontend
    Railway/Render  -> hosts the FastAPI backend + Postgres
    Qdrant Cloud (or self-hosted Docker) -> hosts the vector DB
Turns "runs on my laptop" into a real, shareable, deployed link.


THE FULL PICTURE
===================
User logs in (Clerk) -> Next.js UI -> REST request (with JWT) -> FastAPI verifies JWT
                                                                        |
                                          SQLAlchemy (ORM) ---> Postgres (structured data)
                                          Alembic ------------> manages Postgres schema changes over time
                                          qdrant-client -------> Qdrant (semantic search over study chunks)
                                          Inngest -------------> runs long PDF-ingestion jobs in the background
                                          OpenAI --------------> embeds text (for Qdrant) + generates programs

uv -> installs/manages all the Python packages FastAPI, SQLAlchemy, etc. need to even run
